## 二、最容易走偏的六点

### 2.1 Phase 1 范围膨胀

**走偏**：一上来做前端、Qdrant、RabbitMQ、K8s、Grafana。

**正路**：Phase 1 唯一目标是 [里程碑与验收.md](./里程碑与验收.md) Phase 1 表 — **CLI 能问答 + 2 个工具 + Redis + 三服务 gRPC 通**。

> **判断标准**：任何改动若不能推进 Phase 1 checkbox，defer 到 Phase 2/3。

### 2.2 微服务拆得太早、太深

已选定全微服务，但 Phase 1 不需要「完美分布式」。

**Phase 1 最小切片**：

```
CLI / 简单 HTTP
    → agent-core（ReAct + LLM）
        → gRPC → tool-service（search_papers + get_abstract）
    → Redis（会话）
```

**走偏**：先写 Gateway 全套 REST/SSE/device_id，再写 Agent。

**正路**：**先竖切一条链路** — `agent-core` 单进程 + Redis 会话 + 1 个工具能跑通，再拆 gRPC，再加 Gateway。

Gateway 可先只做 health + 转发；Phase 1 前半周可用 CLI 直连 `agent-core`。

### 2.3 ReAct 实现过度设计

**走偏**：第一天就上 JSON Mode 兜底、MiMo/MiniMax 降级、完整 ToolRegistry 抽象。

**正路**：

1. 只接 `deepseek-v4-flash` + **non-thinking**
2. 只实现 Function Calling 主路径
3. `maxSteps=5` + 明确 `error` 事件
4. 降级逻辑放到 Phase 1 最后 1–2 天

### 2.4 工具层贪多

Phase 1 只要 2 个工具，实现「笨但稳」：

| 工具 | Phase 1 范围 |
| ---- | ------------ |
| `search_papers` | **仅 arXiv API**（Semantic Scholar 可 Phase 1 末尾或 Phase 2） |
| `get_abstract` | arXiv ID 直查 |

**走偏**：统一 Paper 模型、缓存层、引用排序算法。

**正路**：返回固定 JSON（`title`, `authors`, `year`, `abstract`, `paper_id`），够 Agent 用即可。

### 2.5 SSE / 前端提前

[API规范.md](./API规范.md) 已规定 `POST /api/v1/chat/stream`，但 **Phase 1 验收是 CLI**，不是 Web。

**走偏**：Phase 1 做 React + SSE 流式渲染。

**正路**：Gateway SSE 放到 Phase 1 末尾或 Phase 2；CLI 打印 `thought/action/observation/answer` 即满足 DoD。

### 2.6 可观测性、K8s 抢跑

Prometheus、Grafana、HPA、Minikube 全部在 **Phase 3**。

Phase 1 最多加结构化日志（`slog`）+ `trace_id`；不要上指标栈。

## 三、第一周施工顺序（推荐）

| 天数 | 交付物 | 验收方式 |
| ---- | ------ | -------- |
| Day 1 | `go.work`、三服务空壳、`proto` 生成代码、`.env.example` | `go build ./...` 通过 |
| Day 2 | `scripts/test_deepseek_fc.go`：单次 FC 调用成功 | 终端看到 `tool_calls` 返回 |
| Day 3 | `agent-core` 内存版 ReAct：1 工具 mock | CLI 输出 `answer` |
| Day 4 | 接入真实 `search_papers`（arXiv） | `"attention mechanism"` 返回 ≥5 条 |
| Day 5 | `tool-service` gRPC + `get_abstract` | agent-core 通过 gRPC 调工具 |
| Day 6 | Redis 会话持久化 | 重启后历史仍在 |
| Day 7 | `docker-compose` 三服务 + gateway health | [里程碑 Phase 1](./里程碑与验收.md) 全项勾选 |

**每日自检**：

> 今天能否用 CLI 问一个问题，并得到比昨天更接近最终形态的答案？

若否，说明当天任务选错了。

### 依赖关系

```
Day 1-2 基础设施
    │
    ▼
Day 3-4 Agent 闭环（单进程 + mock/真实工具）
    │
    ▼
Day 5 工具 gRPC 化
    │
    ▼
Day 6-7 会话持久化 + docker-compose
```

## 四、开工前技术验证（约半天）

在写业务代码前，用**独立脚本**验证外部依赖：

### 4.1 DeepSeek Function Calling

- 模型：`deepseek-v4-flash`
- 配置：`extra_body: {"thinking": {"type": "disabled"}}`
- 带 1 个 tool definition，确认返回 `tool_calls`

### 4.2 arXiv API

- 用 `query=all:attention` 能拿到结果
- 确认限流策略（加简单 sleep/retry）

### 4.3 Redis

```bash
docker run -d -p 6379:6379 redis:7-alpine
```

- 读写 `session:{id}:messages` 列表

三项都通再动 `agent-core`；否则易把环境问题误判为架构问题。

## 五、代码结构硬约束

动工时遵守以下规则，降低后期重构成本：

### 5.1 共享类型只放一处

| 包路径 | 内容 |
| ------ | ---- |
| `pkg/agent/events.go` | `StepEvent` 等 Agent 事件类型 |
| `pkg/paper/types.go` | 论文结构体 |

三服务共同引用，**禁止各服务复制 struct**。

### 5.2 服务职责边界

| 规则 | 说明 |
| ---- | ---- |
| Agent 不直接调 arXiv | 所有工具经 `tool-service` gRPC |
| LLM 只在 agent-core | `tool-service` 不做 LLM 调用 |
| Embedding 只在 pdf-worker | Phase 2 再引入，Phase 1 不涉及 |
| 配置全部环境变量 | 端口、模型名、maxSteps、TTL 不写死 |

### 5.3 目录结构（与仓库根一致）

```
Scholar-Agent/
├── gateway/
├── agent-core/
├── tool-service/
├── pdf-worker/          # Phase 2
├── frontend/            # Phase 2
├── pkg/                 # 共享类型
├── proto/               # gRPC 定义
├── scripts/             # 验证脚本
├── deploy/
└── docs/
```

详见 [技术设计.md](./技术设计.md) §1.3。

## 八、风险与缓解（动工视角）

| 风险 | 缓解 |
| ---- | ---- |
| 三微服务一周工期紧 | 先 CLI 直连 agent-core，Gateway 最后接 |
| DeepSeek thinking 默认增延迟 | Agent 循环强制 non-thinking（见 [产品需求.md](./产品需求.md) §4.3） |
| arXiv 限流 | 简单 retry + 主路径只用 arXiv |
| 过早做前端 | Phase 1 以 CLI 验收为准 |
| proto 与实现漂移 | 先落盘 `proto/`，再生成 Go 代码 |

## 十、一句话总结

文档已定义「建什么」；动工关键是锁定 **Phase 1 竖切路径**：

**先 agent-core 单链路 → 再 gRPC → 再 Redis → 最后 compose**。

凡是不服务于 CLI 问答闭环的能力（前端、RAG、K8s、监控），一律不许插队。
